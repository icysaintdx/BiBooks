# BiBooks 桌面版完整业务流程与功能说明

更新时间：2026-06-17  
适用目录：`D:\businessPPA\BiBooks-desktop`  
当前定位：先完成可用、可测、可商业演示的本地桌面版；平台版协同、权限、审计、多租户和集中模型服务后置。

## 1. 核心结论

当前桌面版已经不是单页技术方案工具，已经具备项目管理、技术方案、商务标、报价、知识库、查重、废标项、投标机会、竞争分析、合规检查、交付检查和版本审阅的基础框架。

但它还没有完成“完整投标文件生产系统”的最后一层：完整标书合成器、版式模板中心、资质/业绩附件库、OCR 识别校验、报价 Excel 核算脚本、商务标附件自动插入、导出前总控校验。

因此当前判断如下：

- 技术方案：主流程已基本成型，可继续测试和修复。
- 商务标：目前是结构化内容草稿/预览清单，不是最终递交标书。
- 报价管理：本地计算和持久化已具备基础能力，已修复旧库缺列导致的加载失败风险，但还缺 Excel 导入、预算/控制价校验和最终合成注入。
- 项目管理：已具备项目选择、新建、密码、回收站、项目目录和招标文件复制逻辑，但仍需继续检查 UI 细节。
- 最终导出：当前更偏技术方案 Word 导出，还不是“技术标 + 商务标 + 报价 + 资质图片 + 业绩证明 + 封面目录 + 统一版式”的完整标书合成。

2026-06-17 更新：当前已经具备完整导出雏形，技术方案导出归档页会尝试拼接最新商务材料和报价数据，生成“完整投标文件”Word 大纲。但它仍是雏形，不等同于正式完整标书合成器，仍缺企业资质/业绩图片自动插入、版式模板中心、封面目录模板、PDF 导出和导出前总控审批。

## 2. 正确业务闭环

一个投标项目应按下面顺序执行：

1. 启动软件
   打开后默认进入项目管理，不自动进入某个招标文件或技术方案。用户必须选择或新建一个项目。

2. 新建项目
   用户选择招标文件，系统只在点击“新建项目”后创建项目目录、复制招标文件、写入数据库。项目名称可从招标文件名自动带入，并删除“招标文件”等无效字样，用户可再修改。

3. 进入项目
   当前项目成为所有菜单的数据上下文。技术方案、商务标、报价、合规、查重、竞争分析等都必须对齐当前项目。加密项目每次启动都要重新输入密码。

4. 招标文件解析
   上传/绑定招标文件后解析项目名称、采购范围、评分规则、资格要求、废标条款、工期、交付要求、格式要求、报价要求、附件清单。

5. 解析质量检查
   检查是否乱码、表格是否丢失、扫描件是否需要 OCR、页码是否连续、章节是否缺失、关键条款是否提取完整。问题进入交付检查/修复中心。

6. 目录生成
   根据招标要求生成技术标和商务标目录。目录只能基于招标文件、企业资料、知识库和人工确认信息，不得由 AI 无依据扩展。

7. 全局事实设定
   确认项目名称、客户名称、投标单位、工期、服务地点、响应范围、关键参数、人员配置、售后承诺等全局事实。缺失项必须标注“需人工补充”。

8. 技术方案生成
   AI 根据招标文件、全局事实、知识库和历史案例生成技术正文。AI 可以组织语言，但不能编造金额、资质、人员、日期、参数、业绩或承诺。

9. 扩写改写
   对技术正文做章节级改写、补强、去重、统一语气和评分点对齐。所有修改进入项目内变更审阅层。

10. 企业资料准备
    在企业知识库维护资质、业绩、人员、财务、合同、验收报告、证书图片等长期资产。后续应通过 OCR 自动识别证书编号、发证单位、有效期，并由人工确认。

11. 商务标生成
    商务标页面应生成商务响应草稿：报价说明、商务条款响应、资质清单、业绩清单、财务情况、售后服务承诺。当前页面输出的 Markdown 是中间层和预览清单，不是最终递交文件。

12. 报价管理
    报价数据由本地脚本/本地数据库处理。税费、折扣、汇总、大写金额、预算偏差、控制价检查、正文金额一致性不能交给 AI 推断。

13. 查重/废标/合规/竞争分析
    这些模块不直接修改最终文档。它们产生问题、建议和风险项，统一进入交付检查。用户确认后再回到对应源页面修复。

14. 交付检查
    汇总技术方案、商务标、报价、查重、废标、合规、竞争分析产生的问题。每个问题必须记录来源、原文、建议、风险等级、处理状态和修订记录。

15. 变更审阅
    软件内部显示新增、删除、修改、移动等 diff。导出 Word/PDF 时只输出最终稿，不带内部审阅标记。

16. 完整标书合成
    后续必须新增完整合成器：读取技术标 Markdown、商务标 Markdown、报价数据、资质/业绩附件、版式模板，统一生成完整 Word/PDF。

17. 导出归档
    导出前显示最终变更摘要、未处理风险、报价校验状态、来源审批状态、附件缺失状态。通过后导出 Word 原稿和 PDF。

## 3. 菜单功能说明

### 项目管理

用途：管理多个投标项目，是软件首页和项目入口。

当前能力：

- 查看项目列表。
- 新建项目。
- 选择招标文件。该控件是文件选择动作，不应让用户手输路径。
- 自动复制招标文件到项目目录。
- 项目名自动建议。
- 项目备注。
- 可选项目密码。
- 进入项目。
- 改名/修改密码。
- 删除到回收站。
- 恢复项目。
- 立即销毁。

页面输入和按钮：

- “选择招标文件”：打开本地文件选择器。选中文件后只缓存待创建信息，不立即创建项目；项目名称为空时自动带入招标文件名，并清理“招标文件”等无效字样；项目名称已有内容时询问是否覆盖。
- “项目名称”：正式项目名称。必填，可以由招标文件名自动带入，也可人工修改。
- “招标文件”：只读显示已选择文件名。文件副本会在点击“新建项目”后复制到项目目录。
- “访问密码（可选）”：为当前项目设置本地访问密码。设置后每次启动软件都需要重新输入密码才能进入。
- “项目备注”：记录客户、截止时间、联系人或其他内部说明。
- “新建项目”：此时才建立项目目录、复制招标文件、写入数据库。
- “进入项目”：将该项目设为当前项目，并恢复上次所在模块；如果上次模块是项目管理，则进入技术方案。
- “改名/密码”：修改项目名称、备注或重设访问密码。
- “删除”：二次确认后进入回收站，7 天后计划销毁。
- “回收站”：查看已删除项目。
- “恢复”：把项目从回收站恢复到正常列表。
- “立即销毁”：跳过 7 天等待，彻底删除，需再次确认。

应继续完善：

- 删除必须二次确认，当前已有“输入项目名称确认”的弹窗，但要继续测试阻塞状态。
- 回收站 7 天后销毁逻辑已存在，页面已经显示删除时间和计划销毁时间，仍需检查自动清理任务是否按预期执行。
- 新建按钮卡顿/输入框短时间不可点需要继续复测。
- 所有项目目录结构需要标准化：`source`、`technical`、`commercial`、`pricing`、`attachments`、`exports`、`reviews`。

可能问题：

- 未选择项目时进入其他菜单，应拦回项目管理。
- 加密项目未解锁，应禁止读取项目数据。
- 招标文件被外部删除，不影响项目内副本。

### 技术方案

用途：制作技术标主体内容。

主流程：

- 上传招标文件。
- 招标文件解析。
- 目录生成。
- 全局事实。
- 正文生成。
- 扩写改写。
- 交付检查。
- 导出归档。

当前能力：

- 支持本地/外部解析链路。
- 支持解析质量摘要。
- 支持目录和正文生成。
- 支持来源标注。
- 支持交付检查入口。
- 支持技术正文问题回写。
- 支持 Word 导出技术方案。

应继续完善：

- 表格解析需保存结构化表格元数据，而不只是 Markdown 表格。
- 扫描件需要 OCR 质量提示。
- 技术方案导出应纳入完整标书合成器，而不是单独作为最终结果。
- 扩写后的问题回写已接入基础链路，但需要端到端人工验证。

页面按钮和流程：

- 顶部步骤：上传招标文件、招标文件解析、目录生成、全局事实设定、生成正文、扩写改写、交付检查、导出归档。
- “上一步/下一步”：按依赖推进。未上传招标文件、解析未完成、目录未生成或全局事实未完成时，下一步禁用并显示原因。
- “重置”：清空当前项目技术方案进度，需确认。
- “首页”：回到上传招标文件步骤，不退出项目。
- “版本”：打开版本管理，查看、保存、对比、恢复历史版本。内部 diff 只在软件内显示。
- “竞品”：打开技术方案内的竞品分析弹窗，作为内部策略参考。
- “合规”：打开技术方案内的合规检查弹窗，问题应回写交付检查。
- “协同”：当前桌面版不作为生产级协同能力，只能作为后续平台版前置实验或本地辅助能力。
- “导出 Word”：在生成正文步骤可导出当前技术方案；在导出归档步骤可尝试导出完整投标文件大纲。
- “继续扩写”：从生成正文进入扩写改写。

交付检查中的操作：

- “返回正文”：处理技术章节缺失、待补充、来源不明等问题。
- “去报价管理”：处理报价为空、金额异常、项目名称缺失等问题。
- “去商务标”：处理商务基本信息、资质、业绩、售后、财务材料问题。
- “去查重/合规”：处理查重和合规结果。
- “应用低风险修复”：只允许低风险、目标明确的修订自动应用到章节正文；商务、报价、资质和金额类问题必须人工确认。

可能问题：

- 招标文件解析不完整，进入解析质量检查。
- AI 生成内容无依据，进入来源审批。
- 正文缺章节，进入交付检查。
- 表格显示像 Markdown，说明编辑层仍是中间层；导出层应转真实 Word 表格。

### 商务标

用途：生成商务响应内容草稿，不是单独最终递交标书。

当前页面内容：

- 项目名称。
- 投标单位。
- 报价类型。
- 报价说明。
- 商务条款响应。
- 资质证明清单。
- 业绩证明清单。
- 财务情况。
- 售后服务承诺。
- 商务标 Markdown 报告。

正确定位：

- Markdown 是中间层，用于结构化、预览、审阅和后续合成。
- 商务标最终应由完整合成器转成 Word 章节。
- 资质和业绩图片应从企业知识库自动插入最终标书。
- 报价真实金额应从报价管理模块注入，不应让 AI 编写。

页面输入和按钮：

- “基本信息”步骤：
  - “项目名称”：默认从当前项目带入，可人工修改。
  - “投标单位”：当前仍需手填，后续应从企业知识库读取。
  - “报价方式”：选择总价包干、单价合同等报价模板。
  - “下一步”：项目名称和投标单位为空时不允许进入材料配置。
- “材料配置”步骤：
  - “总金额”：当前可填，但正式逻辑应从报价管理读取，不建议作为主入口长期手填。
  - “添加费用项”：生成商务草稿中的费用说明，后续应改为引用报价单。
  - “资质证明材料”：证书编号、有效期、状态当前可手填；后续应改为企业知识库选择 + OCR 识别 + 人工确认。
  - “业绩证明材料”：当前可添加业绩摘要；后续必须绑定合同、验收报告等附件。
  - “售后服务与财务状况”：填写质保期、响应时间、银行授信、信用评级。
  - “生成商务材料草稿”：生成并保存 Markdown 中间层，同时把缺报价、缺资质、缺业绩等问题写入修复任务。
- “草稿结果”步骤：
  - “复制草稿”：复制 Markdown 中间层，用于调试或人工查看，不是最终递交文件。
  - “重新生成”：回到基本信息重新开始。
  - “修改配置”：回到材料配置修改。
- “历史草稿”：
  - “加载”：恢复历史商务材料草稿。
  - “删除”：删除该草稿记录。

应继续完善：

- 商务标页面改成“商务材料编排器”，减少手输。
- 资质从企业知识库勾选，不在项目里重复录证书信息。
- 业绩从企业知识库勾选，并绑定合同、验收报告、评价、发票等证明材料。
- 商务标生成后进入交付检查，缺资质、缺业绩、报价缺失都必须成为修复项。

### 报价管理

用途：本地维护报价单、明细、税率、折扣和汇总。

当前能力：

- 报价单列表。
- 新建报价单。
- 保存报价单。
- 删除报价单。
- 添加/编辑/删除报价项。
- 本地计算税前小计、优惠、税额、含税合计。
- 复制 Markdown 中间层。
- 保存后生成修复任务。
- 已修复旧数据库缺列导致“加载报价单失败”的风险。
- 已修复历史坏 JSON 数据导致整页加载失败的风险。

页面输入和按钮：

- “报价单”下拉：选择当前项目已保存的报价单。
- “新建报价单”：创建一个新的本地报价单草稿，默认项目名称从当前项目带入。
- “保存报价单”：写入本地 SQLite，并根据报价完整性生成修复任务。
- “项目名称”：报价单名称/项目绑定名称，应优先与当前项目一致。
- “税率”：免税、3%、6%、9%、13%。
- “优惠率”：按百分比录入，系统本地计算优惠金额。
- “添加报价项”：打开明细弹窗。
- “复制 Markdown 中间层”：复制报价结构文本，只用于中间层或检查，不是最终报价表。
- “删除报价单”：确认后删除当前报价单。
- “添加/编辑报价项弹窗”：
  - 分类、名称、规格型号、单位、数量、单价。
  - 小计由本地代码实时计算。
  - 名称为空时不能保存。

异常处理：

- 加载失败：现在会尽量兜底为空报价单，同时提示错误；旧库缺列和坏 JSON 已做兼容。
- 明细为空：保存后进入交付检查，标记为严重问题。
- 数量小于等于 0 或单价异常：进入修复任务，不能由 AI 猜测。

边界：

- 报价不传 AI。
- AI 不得猜测单价、数量、总价、预算、控制价。
- Markdown 只是中间层；最终金额由本地数据在合成阶段注入。

应继续完善：

- Excel 导入/导出。
- 控制价/预算对比。
- 漏项/重复项检查。
- 大写金额核验。
- 正文金额与报价表一致性检查。
- pandas + openpyxl 确定性核算脚本。

### 知识库

用途：通用素材、行业模板、历史案例、可复用文本。

当前定位：

- 更偏通用资料库。
- 可为技术方案和商务标提供参考。

应继续完善：

- 行业模板按工程、政府采购、信息化、运维服务、设备采购等分层。
- 每条知识都要有来源、适用范围、风险边界。
- 知识库内容只能作为参考，不能覆盖招标文件事实。

### 企业知识库

用途：企业专属资料库，是商务标和资质附件的核心来源。

应包含：

- 企业基础信息。
- 营业执照。
- 体系认证。
- 行业资质。
- 人员证书。
- 财务报表。
- 纳税/社保/信用证明。
- 业绩合同。
- 验收报告。
- 客户评价。
- 资质和业绩扫描件。

当前不足：

- 还不是完整资质/业绩资产管理器。
- OCR 自动识别证书编号、有效期、发证单位尚未落地。
- 附件图片和最终商务标自动插入链路尚未完成。

### 标书查重

用途：检查目录、正文、图片、表格、表达重复或多项目复用风险。

正确流程：

- 上传/读取当前项目标书内容。
- 检查重复表达。
- 生成问题项。
- 回写交付检查。
- 用户回到技术方案或商务标修改。
- 再次复核。

当前能力：

- 查重分析基础页面存在。
- 查重结果可进入修复任务链路。

应继续完善：

- 项目级数据隔离要继续核查。
- 查重结果和具体章节定位要更精确。
- 修改建议不能直接覆盖正文，必须审阅后应用。

### 废标项检查

用途：检查硬性废标风险。

检查对象：

- 资格条件。
- 签章盖章。
- 文件格式。
- 报价范围。
- 工期响应。
- 付款条款响应。
- 附件完整性。
- 法定代表人/授权委托。

应继续完善：

- 与当前项目解析结果绑定。
- 与企业资质库绑定。
- 与完整导出前检查绑定。

### 投标机会

用途：售前线索和投标决策管理，不是制作标书主流程的一部分，也不是投标文件组成部分。

正确用途：

- 记录机会来源、客户、预算、截止时间、状态、决策评分。
- 评估这个机会是否值得投入人力、时间和报价资源。
- 决定参与后，再到“项目管理”中新建正式投标项目。

所属阶段：

- 机会阶段：投标机会。
- 准备阶段：知识库、企业知识库、版式模板、企业资料。
- 制作阶段：技术方案、商务标、报价。
- 检查阶段：查重、废标检查、合规检查、竞争分析、交付检查。
- 交付阶段：预览编辑、完整合成、导出归档。

页面输入和按钮：

- “列表/看板/日历”：切换机会视图。看板现在是多列自适应布局。
- “新增机会”：录入项目名称、招标编号、客户、预算、截止日期、来源和说明。
- “日历”：按机会的截止日期、投标日期、开标日期生成日程。如果没有填写这些日期，日历会显示为空状态。
- 列表项点击：查看机会详情。
- “状态流转”：在已发现、分析中、决策中、投标中、中标、未中标、已取消、已放弃之间切换。
- “决策因素评分”：按项目匹配度、竞争程度、利润潜力、资源可用性、客户关系、风险水平等因素评分。
- “分析并生成建议”：根据本地评分生成参与建议。
- “竞争对手”：录入竞品名称、优势、劣势、胜率，用于内部决策，不直接写入正式标书。
- “删除”：删除当前机会记录。

边界：

- 投标机会是线索管理，不是项目制作入口。决定投标后，应在项目管理中新建正式投标项目。
- 竞争信息若来自互联网或推断，必须进入来源标注和人工确认流程，不能直接写入正式文档。

### 竞争分析

用途：分析评分权重、竞品优势、差异化策略。

边界：

- 可作为内部策略参考。
- 不能把未经证实的竞品推断直接写入正式标书。
- 若使用互联网信息，必须标注来源并等待人工确认。
- 分析结果应生成内部建议或修复任务，不能直接覆盖技术正文。

### 合规检查

用途：检查最终内容是否符合招标文件和内部规则。

正确流程：

- 从当前项目读取技术标、商务标、报价、附件状态。
- 输出问题项。
- 回写交付检查。
- 修复后再次检查。

当前能力：

- 页面和结果存储基础链路存在。
- 已接入修复任务回写。

应继续完善：

- 合规检查必须接入完整标书合成前置校验。
- 合规问题要能定位到章节、表格、附件或报价字段。

页面规则：

- 输入内容应尽量从当前项目解析结果、技术正文、商务材料和报价单读取。
- 发现问题后进入修复任务，用户回到源页面处理。
- 修复后再次运行合规检查，不能只在结果页手动标记通过。

### 设置

用途：模型、解析、开发模式等配置。

当前能力：

- 支持 OpenAI-compatible 模型。
- 支持本地模型服务，例如 Ollama、LM Studio、llama.cpp server、vLLM。
- 本地模型空 API Key 请求头问题已修复。

应继续完善：

- 本地模型连接向导。
- 模型能力检测。
- OCR 模型配置。
- 脱敏规则配置。
- 提示词边界配置。

设置页边界：

- 云端模型可以用于文本组织、目录建议、风险提示，但不得接收真实报价、控制价、预算、敏感证件、账号等数据。
- 本地模型适合涉密客户，但桌面版短期只做连接配置和检测，不把模型权重作为强依赖打包。
- OCR 模型后续应作为企业知识库的本地能力，不应把资质证书图片上传到外部服务。

## 4. OCR 与资质材料逻辑

资质证书和业绩材料不应长期靠项目页手输。正确方案是企业知识库一次维护、项目里选择引用。

后续 OCR 流程：

1. 上传证书或合同/验收报告扫描件。
2. 本地 OCR 识别证书名称、编号、发证单位、发证日期、有效期、项目名称、合同金额、甲方名称等。
3. 系统做格式校验和过期校验。
4. 识别不确定、格式异常、已过期，页面高亮并显示信息提示。
5. 未人工确认前不能作为“已完成资质”正常入库。
6. 项目商务标选择该资质或业绩后，最终合成器自动插入文字摘要和证明图片。

当前状态：

- OCR 资质库尚未完成。
- 图片自动插入最终标书尚未完成。
- 当前商务标页面只形成文字清单/草稿。

## 5. MD 中间层说明

系统内部允许使用 Markdown，因为它适合做结构化文本中转、审阅、查重和 AI 处理。

但必须明确：

- MD 不是最终递交标书。
- MD 不控制最终字体、页眉页脚、页码、封面、目录、表格样式和图片排版。
- MD 中不应长期保存真实报价敏感数据。
- 正式导出必须由“MD 内容 + 版式模板 + 附件素材 + 本地报价数据”合成 Word/PDF。

## 6. 版式模板中心

这是完整标书合成的基础能力。当前设置页已经有版式模板配置入口，并已支持多模板配置文件的基础数据结构和 UI 操作；后续还要继续把它和项目绑定、导出预览、完整合成器打通。

当前已具备：

- 模板库下拉选择。
- 新建模板。
- 复制模板。
- 删除模板。
- 保存多个模板配置。
- 配置页边距、页眉、页脚、正文、标题、表格、图片规则。
- 支持项目自带字体目录：`client/electron/fonts`，可放入 `.ttf`、`.otf`、`.woff`、`.woff2`。

应支持：

- 页面尺寸。
- 页边距。
- 装订线。
- 页眉文字。
- 页眉 Logo。
- 页脚。
- 页码格式。
- 正文字体、字号、行距、段前段后、首行缩进。
- 一级到四级标题样式。
- 标题编号格式。
- 表格边框、表头底色、单元格边距、跨页规则。
- 图片最大宽度、居中、图注、多图排布。
- 封面模板。
- 目录模板。
- 行业模板保存与切换。

模板保存为项目可引用的配置文件，后续合成完整标书时统一套用。

字段解释：

- 页面尺寸：Word 页面大小，当前以 A4/A3 为主。
- 页边距：上下左右留白，单位毫米。
- 装订线：装订边额外留白。
- 页眉：每页顶部文字或 Logo，可显示项目名、投标单位等变量。
- 页脚：每页底部文字和页码格式。
- 正文字体：普通段落的默认字体、字号、行距和首行缩进。
- 标题层级：一级到四级标题的字体、字号、加粗、对齐和编号样式。
- 表格规则：报价表、资质表、业绩表的表头底色、边框、跨页和重复表头。
- 图片规则：资质证书、合同扫描件、方案配图的宽度、对齐和题注。

字体说明：

- 常用系统字体可以直接填写名称，如宋体、仿宋、黑体、楷体。
- 公文或招投标专用字体可以放入 `client/electron/fonts`。
- 后续正式合成器应优先使用模板指定字体；如果本机没有该字体，则提示用户并回退到可用字体。

## 6.1 合并前预览编辑

正式完整合成器前必须有“合并预览编辑”阶段。

正确能力：

- 合并前预览完整投标文件结构：技术标、商务标、报价文件、资质附件、业绩附件、封面、目录。
- 预览时仍可编辑文字。
- 可选择某一段调用 AI 改写，但 AI 必须基于已有文件、知识库或人工输入，不能凭空补事实。
- 可在某一页前、页后或章节内插入文字、图片或表格。
- 插入图片应复制到项目附件目录，并记录来源。
- 插入表格应是真实结构化表格，不应只是 Markdown 管道文本。
- 所有预览阶段修改都进入项目内变更审阅层，导出正式 Word/PDF 时只输出最终稿。

## 7. 变更审阅层

项目内应分三层：

- 原始层：招标文件解析、AI 生成、本地脚本结果、历史案例、知识库、人工输入原文。
- 修订层：每次修改形成 patch，记录新增、删除、替换、移动。
- 发布层：最终导出的 Word/PDF，只取当前生效版本。

当前状态：

- 修复任务和 patch 字段已经落地。
- 交付检查已开始汇总技术方案、商务标、报价、查重、合规等问题。
- 技术正文低风险修订可自动应用。
- 商务标、报价、查重、合规仍应以人工审阅为主。
