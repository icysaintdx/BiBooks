## 背景 / 问题描述

阶段一（v0.13.0 统一编排骨架版）已落地统一编排骨架、状态持久化修复、商务标 / 报价招标解析联动、大纲招标格式优先路由基础渠道。阶段二原规划 4 项 TODO，本轮按用户决策只做低风险 3 项，商务标 / 报价"动态字段 schema 化"暂缓到单独一轮（当前表单是纯硬编码 useState + JSX，schema 化属中高难度重写，混入本轮会拉高回归风险）。

本轮 3 项遗留问题：

1. **编排卡片状态写死**：编排区 `BidOrchestrationBand` 的商务标 / 报价两张卡片状态恒为硬编码 `'todo'`（待处理），未反映当前项目下两块的真实完成度。
2. **大纲格式渠道只到一级**：`tenderStructureWorkflow` 的一级标题已来自招标 `bidFileStructure`，但补全二三级目录时调用的是通用 `generateChildren`，未携带 `bidFileStructure`，二三级目录仍靠技术评分自由发挥，未贴合招标对各条目的组成 / 格式规定。
3. **导出只有软提示**：导出前一致性校验只有 `warnings` 软提示与"缺商务标 / 缺报价"二元拦截，未按《完整标书合成与版式模板方案》§8 规定的三级（致命 / 重大 / 普通）拦截，致命问题无法禁止导出正式稿。

核心原则（用户硬性要求，贯穿本轮）：始终以招标文件要求 / 示例 / 说明为最高优先级，硬编码逻辑只作兜底，冲突时以招标文件为准；渐进叠加，不破坏现有流程。

## 解决方案

### 1. 编排区进度联动（前端，纯加法）

`TechnicalPlanHome.tsx` 新增 `commercialBidStatus` / `pricingStatus` 状态（复用 `OrchestrationBlockStatus`，初值 `'todo'`），新增 `useEffect`（依赖 `[currentProject?.id, state.outlineData]`，目录就绪后触发）：

- 调 `window.yibiao.commercialBid.list()` 与 `window.yibiao.pricing.list()`（均 `.catch(()=>[])` 容错，且已按当前项目自动过滤），客户端再按 `bidProjectId` 过一道并用 `cancelled` 标志防竞态；
- 商务标：无记录→`'todo'`；有记录但无 `result`/`report`→`'active'`（进行中）；任一有 `result`/`report`→`'ready'`（可开始 / 已就绪）；
- 报价：无 `items`→`'todo'`；有 `items` 但含税合计 `<=0`→`'active'`；含税合计 `>0`→`'ready'`；
- 编排卡片 business / pricing 两块的 `status` / `statusLabel` 由硬编码改为读上述状态。

### 2. 大纲格式渠道深化（后端，受 try/catch 兜底保护）

`outlineGenerationTask.cjs` 新增 `generateTenderStructureChildrenMessages`（仿 `generateAlignedChildrenMessages`，把招标 `bidFileStructure` 作为最高优先级输入 + 当前一级条目标题，要求二三级目录贴合招标对该条目应包含内容与编制格式的规定）与 `generateTenderStructureChildren`（与 `generateChildren` 同构，改用新消息构造器）。`tenderStructureWorkflow` 循环中把 `generateChildren` 换成 `generateTenderStructureChildren`，`payload.bidFileStructure` 直接透传。外层 `tenderStructureWorkflow` 仍由 try/catch 包裹，异常回退 aligned/free，不阻塞主流程。本轮不改 `bidFileStructure` 解析项 prompt（第六章范本表格正文抽取留后续）。

### 3. 导出三块一致性硬校验（前端，升级既有结构）

`TechnicalPlanHome.tsx` 新增 `ExportIssueLevel`（`fatal`/`major`/`minor`）与 `ExportIssue` 类型，`buildProjectExportPreview` 产出结构化 `issues: ExportIssue[]`：

- **致命**：缺技术正文（目录为空 / 无任一章节）、报价明细为空、商务标缺失 → `canExport=false`，禁止导出正式稿；
- **重大**：报价含税合计 `<=0`、商务标无资质 / 无业绩、技术正文章节过少（<2）→ 允许导出但需确认；
- **普通**：现有来源 / 格式类软提示统一归入 minor。

`canExport = issues.every(i => i.level !== 'fatal')`，并保留 `warnings`（由 `issues` 派生）兼容现有 UI 与导出进度弹窗。`ExportArchivePage` 按 level 分组渲染（致命 / 重大 / 普通），导出按钮禁用沿用 `exportBlocked={!canExport}`；存在重大警告时点击导出弹 `window.confirm` 二次确认。

## 代码示例

编排进度联动推导（`TechnicalPlanHome.tsx` 节选）：

```tsx
const latestBid = scopedBids[0] as any;
if (!latestBid) {
  setCommercialBidStatus('todo');
} else if (latestBid.result || latestBid.report) {
  setCommercialBidStatus('ready');
} else {
  setCommercialBidStatus('active');
}
```

导出三级分级（`TechnicalPlanHome.tsx` 节选）：

```tsx
const fatalIssues = issues.filter((issue) => issue.level === 'fatal');
const canExport = fatalIssues.length === 0;
return {
  markdown: outlineToMarkdown(projectExport.outline),
  issues,
  warnings: issues.map((issue) => issue.message),
  bidderName: bid?.companyName || '',
  canExport,
  blockReason: fatalIssues.length ? fatalIssues.map((issue) => issue.message).join(' ') : '',
};
```
