## 背景 / 问题描述

桌面版投标书生成流程在结构上存在三类问题：技术方案 / 商务标 / 报价三块脱节、技术方案步数跨板块切换丢失、商务标 / 报价 / 大纲为硬编码固定模式（未读取招标文件解析结果）。本次为阶段一改造，建立"统一目录驱动"的三块编排骨架，并修复状态持久化与行业字段透传。

核心原则（用户硬性要求）：始终以招标文件要求 / 示例 / 说明为最高优先级，硬编码逻辑只作兜底，冲突时以招标文件为准。

## 解决方案（阶段一已落地概览）

1. **统一编排骨架**：以 `TechnicalPlanHome.tsx` 为编排宿主，目录生成完成（`state.outlineData` 非空）作为分水岭，在 step 内容上方渲染编排区 `BidOrchestrationBand`（技术正文 / 商务标 / 报价三张状态卡片）。技术正文留在本页流程，商务标 / 报价通过 `onNavigateSection` 跳转。渐进叠加，原单线 step 流程完全保留。
2. **状态持久化修复**：前端 `validSteps`、后端 `isValidStep` 补齐 `delivery-check`、`export-archive`，跨板块切换不再丢步。
3. **industryCode 透传修复**：`applyPartial` 处理 `industryCode` / `industryName` 并写入 `technical_plan_meta`，`loadTechnicalPlan` 回读，大纲生成可加载行业知识。
4. **商务标 / 报价联动**：`AppRouter.tsx` 传入 `currentProject`；通过 `shared/utils/tenderLinkage.ts` 从招标解析结果填充表单初值（缺失回退硬编码默认）；商务标持久化 / 还原表单；两块持久化 `bid_project_id`。
5. **大纲招标格式优先**：新增 `bidFileStructure` 解析项；`outlineGenerationTask.cjs` 读招标原文 + 该解析项；招标规定组成 → 走 `tenderStructureWorkflow`，未规定 → 回退评分对齐 / 自由生成，异常 try/catch 强制回退。
