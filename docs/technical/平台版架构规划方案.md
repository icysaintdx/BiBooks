**日期**: 2026-06-15 10:00

**责任人**: BiBooks 开发团队

## 一、背景与转折点

### 1.1 当前版本（master）

BiBooks master 分支为 Electron 桌面单机版：单用户、SQLite 存储、无认证、AI 通过云端 API 或本地 Ollama 实现。已完成 14 步标书生成流水线核心功能，脱敏中间件（desensitization.cjs）已实现金额/报价数据自动替换为占位符后送入 AI。

### 1.2 关键转折点

用户提出新需求：

> 类似一个平台，支持多用户同时调用 API 和本地模型，同时进行不同操作。
> 需要区分用户权限，像 GitHub 一样有版本控制，显示谁在什么时候动了什么。
> 当有用户正在修改时，其他用户只能查看，以免冲突。

**架构转变：**

| 维度 | 桌面版（master） | 平台版（platform-v2） |
|------|-----------------|----------------------|
| 运行形态 | Electron 本地 | 服务器 + 浏览器 |
| 存储 | SQLite 单机 | PostgreSQL 服务端 |
| 认证 | 无 | JWT 多用户 |
| 并发控制 | 无冲突（单人） | 编辑锁防冲突 |
| 历史版本 | 无 | 版本控制（谁改了什么） |
| 部署 | 安装包 | Docker Compose |

这是本项目的**关键转折点**：从桌面单机工具 → 团队协同平台。

## 二、分支策略

不新建仓库，使用分支隔离：

| 分支 | 定位 | 状态 |
|------|------|------|
| master | 桌面单机版（稳定维护） | 持续迭代，不停止 |
| platform-v2 | 平台多用户版（本分支） | 新开发，架构演进 |

**理由：** 两者共享 90% 业务逻辑（标书生成、AI 调用、脱敏、文件解析），分支可持续同步 master 功能迭代，避免维护两套代码库。

## 三、原始设计要点（来自 docs/Think.txt）

### 3.1 三层能力分工

根据项目原始设计文档（docs/Think.txt）的核心结论：

| 层级 | 技术 | 职责 | 工作量占比 |
|------|------|------|-----------|
| 第一层 | Python 脚本（python-docx/pandas/openpyxl） | 报价计算、格式处理、Word 填充 | 65% |
| 第二层 | 本地 7B 量化模型（via Ollama） | 招标文件结构化提取、轻量分类 | 辅助 |
| 第三层 | 云端 API（DeepSeek/Qwen） | 定制化方案撰写、评分响应 | 35% |

### 3.2 数据隔离原则（核心不可动摇）

报价/金额/涉密数据 → Python 脚本处理 → 永远不送 AI

通用文案内容 → desensitization.cjs（脱敏） → AI → 本地脚本合并最终标书

### 3.3 模型选型（已定稿）

| 优先级 | 模型 | 中文适配度 | 最低配置 |
|--------|------|-----------|---------|
| 首选 | DeepSeek-7B-Chat 4bit GGUF | 95%（公文语感极强） | 16G RAM，无需 GPU |
| 次选 | Qwen2.5-7B-Chat 4bit GGUF | 93%（中文理解最强） | 16G RAM，无需 GPU |
| 备选 | Llama3-8B 4bit | 85%（通用能力强） | 16G RAM，无需 GPU |

排除：1B/2B（逻辑混乱，质量差）；13B+（成本翻倍，无额外收益）。

部署方式：Ollama（OpenAI 兼容 API，Docker 原生支持，GGUF 一键拉取）。

### 3.4 产品形态路线（三条线并行）

| 版本 | 用户群 | AI 方案 | 数据存储 |
|------|--------|---------|---------|
| V2.0 云端脱敏版（master 主线） | 90% 小微企业 | 云端 API + 强制脱敏 | SQLite 本地 |
| V2.5 本地离线版 | 涉密企业/国企 | 本地 7B（完全断网） | 本地 |
| 平台版（platform-v2） | 团队/工作室 | 云端 API + 服务端本地模型 | PostgreSQL 服务器 |

## 四、平台版技术架构

### 4.1 整体架构

```
用户浏览器
    |
    | HTTPS / WebSocket
    |
Nginx（反向代理 :80）
    |
    +-- 静态资源 --> client/dist（React Web App）
    |
    +-- /api/*   --> Node.js API Server（server/ :9800）
                        |
                        +-- Express 路由层
                        +-- WebSocket 实时层
                        +-- JWT 认证中间件
                        +-- RBAC 权限控制
                        +-- 编辑锁服务
                        +-- 操作历史/版本控制
                        |
                        +-- PostgreSQL（主数据库）
                        |       +-- users
                        |       +-- tasks
                        |       +-- edit_locks
                        |       +-- operation_history
                        |       +-- content_versions
                        |
                        +-- Ollama（本地 7B 模型 :11434）
                        |
                        +-- Python 脚本（报价/Word 生成）
```

### 4.2 前端改造策略

抽象 API 适配层，使同一套组件既能在 Electron 中运行，也能在浏览器中运行：

```typescript
// client/src/hooks/useApi.ts
const isElectron = typeof (window as any).yibiao !== 'undefined';
export const useApi = () => isElectron ? electronAdapter : httpAdapter;
```

| 操作 | 桌面版（IPC） | 平台版（HTTP） |
|------|-------------|--------------|
| 文件上传 | window.yibiao.files.upload | POST /api/v1/files/upload |
| 任务列表 | window.yibiao.tasks.list | GET /api/v1/tasks |
| AI 生成 | window.yibiao.ai.stream | WebSocket 流式推送 |
| 配置读取 | window.yibiao.config.get | GET /api/v1/config |

## 五、数据库设计（PostgreSQL）

### 5.1 用户与权限

```sql
CREATE TABLE users (
    id            SERIAL PRIMARY KEY,
    username      VARCHAR(50) UNIQUE NOT NULL,
    password_hash VARCHAR(255) NOT NULL,
    role          VARCHAR(20) DEFAULT 'member',
    created_at    TIMESTAMPTZ DEFAULT NOW(),
    last_login    TIMESTAMPTZ
);
-- admin: 全权限 + 用户管理
-- member: 读写自己任务，查看所有
-- viewer: 全局只读
```

### 5.2 编辑锁

```sql
CREATE TABLE edit_locks (
    task_id    VARCHAR(50) PRIMARY KEY,
    user_id    INTEGER REFERENCES users(id),
    username   VARCHAR(50),
    locked_at  TIMESTAMPTZ DEFAULT NOW(),
    expires_at TIMESTAMPTZ  -- 30 分钟无心跳自动释放
);
```

### 5.3 操作历史与版本快照

```sql
-- 记录谁在什么时间做了什么
CREATE TABLE operation_history (
    id         SERIAL PRIMARY KEY,
    task_id    VARCHAR(50) NOT NULL,
    user_id    INTEGER REFERENCES users(id),
    username   VARCHAR(50),
    action     VARCHAR(50) NOT NULL,   -- edit/generate/export/lock/unlock
    target     VARCHAR(100),           -- 章节名/步骤名
    diff       JSONB,                  -- 修改前/后差异
    snapshot   JSONB,                  -- 完整数据快照
    created_at TIMESTAMPTZ DEFAULT NOW()
);

-- 可回滚的版本快照
CREATE TABLE content_versions (
    id          SERIAL PRIMARY KEY,
    task_id     VARCHAR(50) NOT NULL,
    version_num INTEGER NOT NULL,
    content     JSONB NOT NULL,
    created_by  INTEGER REFERENCES users(id),
    created_at  TIMESTAMPTZ DEFAULT NOW(),
    comment     TEXT
);
```

## 六、多用户协同机制

### 6.1 编辑锁流程

```
用户打开任务
    |
    +-- POST /api/v1/tasks/:id/lock
    |       +-- 成功 --> 进入编辑模式
    |       +-- 失败 --> 显示「张三正在编辑，当前只读」
    |
    +-- 编辑中：每 5 分钟心跳续期，WebSocket 广播锁状态
    |
    +-- 完成：POST /api/v1/tasks/:id/unlock
            +-- WebSocket 广播解锁通知
```

超时：30 分钟无心跳自动释放。admin 可强制解锁任意任务。

### 6.2 操作历史（GitHub 风格时间线）

自动记录所有写操作：

| 操作类型 | 记录内容 |
|----------|---------|
| 编辑内容 | diff（修改前/后文本对比） |
| AI 生成章节 | 内容 + prompt + 模型名称 |
| 导出文件 | 时间、格式、文件名 |
| 获取/释放锁 | 锁定时长 |
| 删除任务 | 删除前完整快照 |

API 接口：
- GET /api/v1/tasks/:id/history — 操作历史列表
- GET /api/v1/tasks/:id/history/:hid/diff — 某次修改详细 diff
- GET /api/v1/tasks/:id/versions — 版本快照列表
- POST /api/v1/tasks/:id/versions/:vid/restore — 回滚到指定版本

### 6.3 WebSocket 实时事件

| 事件 | 触发时机 | 客户端响应 |
|------|---------|-----------|
| task_locked | 某用户获得锁 | 其他用户切换为只读 |
| task_unlocked | 锁释放 | 恢复可申请编辑 |
| task_updated | 内容保存 | 刷新任务内容 |
| task_generating | AI 生成中 | 显示进度条 |
| user_online | 用户进入 | 显示在线用户列表 |

## 七、本地模型部署（平台版）

### 7.1 与桌面版对比

| 对比项 | 桌面版（master） | 平台版（platform-v2） |
|--------|-----------------|----------------------|
| 安装方式 | 用户自行安装 Ollama | Docker Compose 自动部署 |
| 模型拉取 | 用户手动执行 | 容器启动后自动 pull |
| 访问地址 | localhost:11434 | ollama:11434（容器内网） |
| 多用户并发 | 不支持 | 请求队列串行排队 |
| GPU 支持 | 可选 | docker-compose 配置 |

### 7.2 环境变量切换（无需改代码）

```bash
TEXT_MODEL_PROVIDER=ollama          # 使用本地 7B 模型
TEXT_MODEL_PROVIDER=openai          # 使用云端脱敏 API
TEXT_MODEL_NAME=deepseek-r1:7b
OLLAMA_BASE_URL=http://ollama:11434
```

### 7.3 Docker Compose 四服务

已创建的 docker-compose.yml 包含：
- **app**（:9800）：Node.js API 服务
- **ollama**（:11434）：本地 7B 模型，启动后自动 pull
- **postgres**（:5432）：PostgreSQL，数据持久化
- **nginx**（:80）：反向代理（静态 + /api/* + WebSocket 升级）

GPU 加速：docker-compose.yml 已预留注释配置，取消注释即可启用 NVIDIA GPU。

## 十、技术选型说明

**PostgreSQL vs MySQL：**
JSONB 原生支持（历史 diff 存储），行级锁成熟（编辑锁），Supabase/Railway/Neon 托管部署方便。

**Ollama vs 直接集成模型：**
OpenAI 兼容 API 切换零成本；完整模型管理 REST API；Docker 原生 GPU/CPU 切换；GGUF 一键拉取。

**保留 master 桌面版：**
桌面版：个人/小公司（零部署成本）。平台版：团队（需服务器）。两条线覆盖不同规模，共享核心逻辑。
