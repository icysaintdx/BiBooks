# 桌面版当前落地状态与测试清单

更新时间：2026-06-17
适用目录：`D:\businessPPA\BiBooks-desktop`
当前主线：先稳定桌面本地版，再进入平台版重构。

## 方向结论

桌面版先做成可用、可测、可商业演示的本地单机版本。真实多人协同、权限、审计、集中模型服务、多租户和平台化部署放到 `D:\businessPPA\BiBooks-platform` 后续处理。

桌面版中的协同能力不建议作为当前商业主卖点。现有协同更接近本地草稿/辅助能力，生产级多人协同缺少完整的 WebSocket 服务绑定、用户权限、锁、冲突处理和审计闭环。

## 已落地

### 2026-06-17 可用性修复

- 已修复旧库、半迁移库或坏 JSON 数据导致报价页提示“加载报价单失败”的风险。
- `pricing_sheets` 运行期会补齐关键列：`bid_project_id`、`project_name`、`currency`、`tax_rate`、`discount_rate`、`items_json`、`notes`、`summary_json`、`created_at`、`updated_at`。
- 报价单读取时对 `items_json`、`summary_json` 做容错解析，单条历史坏数据不再拖垮整个报价页面。
- 已统一商务标、报价管理、投标机会、竞争分析、合规检查等新增页面的模块容器、卡片、按钮、表格和表单样式。
- 投标机会看板已改成多列自适应网格，不再只横向排成一行；列表、看板、日历三种视图使用统一页面框架。
- 已确认源码中文通过 UTF-8 扫描没有发现用户界面的乱码残留；当前命中的 `documentQuality.cjs` 和 `document-quality-smoke.cjs` 是乱码检测规则和测试数据，不是页面文案。

### 报价本地化

- 新增 SQLite 报价表 `pricing_sheets`。
- 报价页支持本地保存、读取、新建、删除和 Markdown 导出。
- 税率、折扣、明细小计、总价由本地确定性 JS 计算，不交给 AI 推断。
- 已对旧库缺列、历史坏 JSON 做兼容修复，避免页面直接报“加载报价单失败”。
- 已新增 `smoke:pricing`。

仍未完成：Excel 导入导出、控制价/预算对比、漏项检查、费用税费规则引擎、正文金额与报价表一致性检查、`pandas + openpyxl` 报价核算脚本。

### AI 证据边界

- 新增统一提示边界 `client/electron/services/promptPolicy.cjs`。
- 已接入招标分析、目录生成、全局事实、评分分析、正文生成等任务。
- 规则要求 AI 不得无依据猜测或编造项目事实、资质、人员、业绩、金额、工期、地点、设备参数、标准编号、服务承诺。
- 缺失信息必须标为“需人工补充”“原文未提及”“需核验”。
- 报价、预算、税费、统计、数量、金额一致性不允许由 AI 推断，必须走本地确定性计算或人工确认。

### 行业边界

- 新增 `industryPromptPolicy.cjs`。
- 行业知识只能用于结构、风险、表达和常见关注点参考，不能替代招标文件事实。
- 已增强运维服务、政府采购等行业识别与行业知识。
- 已修正全局事实任务，不再允许虚拟项目经理、虚拟日期、虚拟参数、虚拟地点或虚拟承诺。
- 已新增 `smoke:industry-policy`。

### 内部来源标注

- 新增 SQLite 表 `project_source_annotations`。
- 支持来源类型：招标文件、数据库、历史案例、知识库、私有知识库、互联网、人工、AI 生成。
- 互联网来源默认 `requiresApproval=true` 且 `approvalStatus=pending`。
- 正文编辑页新增内部来源/审批面板，可查看来源、风险、摘录，并对待审批来源确认或拒绝。
- 正文生成成功后会自动记录招标文件、全局事实和知识库来源摘要。
- 内部来源标注不进入正式 Word/PDF 导出。
- 已新增 `smoke:source-annotations`。

### 表格链路

- Word 导出已支持 Markdown/GFM 表格转真实 DOCX 表格。
- 新增 `smoke:export-table`，会生成含 Markdown 表格的 DOCX，并检查 `word/document.xml` 中存在 `<w:tbl>`。
- 正式 Word 导出已移除“内容由 AI 生成”等内部标记。

当前判断：用户看到“像 Markdown 图形表格”的问题，主要在解析和编辑阶段仍以 Markdown 文本作为中间形态。导出 Word 已能变成真实表格，但解析阶段还没有保存结构化表格元数据。

### 解析质量检查

- 新增 `documentQuality.cjs`。
- 导入招标文件后会生成 `tenderParseQuality`。
- 已检查字符数、中文比例、标题数量、Markdown 表格数量、表格行数、最大列数、疑似断裂表格行、图片数量、疑似乱码、空内容、扫描件/OCR 风险等。
- 文档分析页已显示解析质量摘要。
- 已新增 `smoke:document-quality`。

### 本地模型

- 已支持 Ollama、LM Studio、llama.cpp server、vLLM 等 OpenAI-compatible 本地模型配置。
- 已修复本地模型空 API Key 被后端拦截的问题。
- 本地模型请求不会再发送空的 `Authorization: Bearer ` 请求头。
- 新增 `smoke:local-model`，使用本地假 OpenAI-compatible 服务验证空 Key 下的 `/models` 和 `/chat/completions` 链路。

### 投标项目工作区

- 新增 `app_projects`、`app_project_meta`、`app_project_snapshots`，数据库版本升至 v10。
- 应用启动后不再只有单份全局技术方案语义，左侧侧边栏新增“当前项目”选择器和“新建”入口。
- 每个项目记录自己的最后打开模块，重新进入或切换项目时会恢复到上次所在模块。
- 技术方案主流程已接入项目快照：切换项目时会保存当前项目的技术方案状态和招标 Markdown，再恢复目标项目自己的技术方案状态。
- 现有单份历史技术方案会被默认项目承接，不会因为新增项目层而消失。
- 新增 `smoke:project-workspace` 和 `smoke:project-workspace-electron`，分别验证内存逻辑和真实 Electron/SQLite v10 迁移。
- 数据库版本升至 v12，报价、商务标、投标机会、竞争分析、合规检查已按当前项目过滤和保存。
- 旧报价、旧商务标、旧投标机会如果没有项目 ID，会在第一次读取时归入当前项目，避免升级后突然不可见。
- 新增 `smoke:project-scoped-modules`，覆盖报价、商务标、投标机会的项目隔离。
- 新增 `project_analysis_records`，竞争分析和合规检查生成成功后会保存输入与结果，切换项目后可恢复当前项目的最新记录。
- 新增 `smoke:project-analysis-store`，覆盖竞争分析、合规检查记录在不同项目之间的隔离。

当前限制：项目级隔离的核心数据链路已经打通，但知识库、查重、废标检查、版本记录仍需要进一步确认是按项目独立，还是作为全局资料库共享。竞争分析、合规检查已保存输入与结果，后续还要补来源标注联动，并尽量从招标解析结果自动带入字段。

### 交付检查与修复任务

- 新增 `project_repair_tasks`，支持来源模块、目标对象、严重级别、建议、patch、状态和处理决策。
- 技术方案章节保存后会按缺正文、正文过短、待补充标记生成或更新修复任务。
- 商务标生成后会把基本信息缺失、报价缺失、必备资质未完成、业绩证明缺失等问题写入修复任务。
- 报价单保存后会把项目名称缺失、报价明细为空、明细数量/单价异常等问题写入修复任务。
- 查重、合规、竞争分析的结果页已经具备回写修复中心的基础链路，仍需人工做端到端验证。

### 完整导出雏形

- 技术方案导出已不是只能导出技术正文：`export-archive` 流程会读取当前项目最新商务材料和报价单，拼入“完整投标文件”导出大纲。
- 商务标和报价目前通过 Markdown 中间层进入导出；导出服务可把 GFM 表格转换成 Word 真表格。
- 当前仍缺企业资质/业绩图片附件自动嵌入、完整封面目录模板、版式模板中心、正式 PDF 导出链路和导出前总控审批。

短期策略：桌面版先做本地模型配置、检测、测试和使用说明；不把 Ollama/Docker/模型权重打进桌面版强依赖。平台版再做服务端/Docker 化的一键部署和集中管理。

## 未落地但必须继续

### 解析增强

- xParse/TextIn 招投标专项结构化抽取尚未接入。
- 表格结构化元数据尚未保存。
- OCR 覆盖率、缺页、页码连续性、表格丢失风险还需要更强的自动报告。
- 小模型质量检查层尚未真正接入，目前已有规则质量检查底座。

### 报价脚本

- `pandas + openpyxl` 脚本层尚未完成。
- 适用场景包括 Excel 报价清单读取、分项汇总、税费折扣、大小写金额、漏项和重复项检查、控制价/预算偏差分析、正文金额与报价表总价一致性核验。

### 脱敏增强

- 现有 AI 服务已有请求脱敏和响应复原基础。
- 后续还要加强折扣率、付款比例、保证金、控制价、预算价、联系人、电话、账号、证书编号等敏感字段规则。
- 开发日志、请求日志、错误日志中的敏感数据也需要专项核查。

### 互联网补充流程

- 当前已经有来源标注和审批底座，但还没有接入完整网络搜索工作流。
- 后续互联网信息只能作为内部候选来源，必须明显标注“互联网来源，未批准不可写入”。
- 经人工同意后才允许作为正文素材，但内部仍保留链接、来源、风险和审批记录。
- 正式文档仍不输出这些内部提醒。

## 下一步人工测试清单

1. 启动桌面版，确认设置页能打开。
2. 在左侧“当前项目”区域创建两个投标项目，分别进入技术方案并切换，确认项目名称和上次模块能恢复。
3. 配置本地 Ollama 或 LM Studio，API Key 留空，点击获取模型，确认能列出模型。
4. 导入一份招标文件，查看解析质量摘要是否合理提示风险。
5. 生成招标分析、目录、全局事实和正文。
6. 在正文编辑页选中具体章节，查看内部来源标注是否出现。
7. 对待审批来源执行确认/拒绝，确认状态能保存。
8. 插入或生成包含表格的正文，导出 Word，确认 Word 中是真实表格。
9. 在报价页新建报价，添加明细，保存后切换页面再回来，确认数据仍在。
10. 在商务标页生成草稿，确认缺资质、缺业绩或缺报价时会进入交付检查/修复任务。
11. 在投标机会页切换列表、看板、日历，确认看板多列显示，左侧菜单不会遮挡内容。
12. 进入技术方案第 7 步“交付检查”，确认可以看到技术、商务、报价、查重、合规来源的问题。
13. 进入技术方案第 8 步“导出归档”，执行完整投标文件 Word 导出。
14. 检查正式导出的 Word 中不出现内部来源标注、审批提醒、“内容由 AI 生成”等内部标记。
15. 如果正文中含表格，打开 Word 确认是可选中单元格的真实 Word 表格，不是 Markdown 管道文本。
