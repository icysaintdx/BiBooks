## 背景 / 问题描述

桌面版投标书生成流程在结构上存在三类问题：

1. **三大块结构脱节**：技术方案（technical-plan）、商务标（business-bid）、报价（pricing）是三个独立的顶层页面。`AppRouter.tsx` 用 `switch(activeSection)` 挂载，每次切换都重新 mount/unmount。改造前商务标、报价页面零 props，完全不知道"当前项目"是谁，也没有任何触发点告诉用户"该处理商务标/报价了"；技术方案则是一条单线"下一步"流程，与另外两块没有结构性连接。

2. **状态丢失 Bug**：技术方案走到第 6 步（delivery-check）或更后，切到商务标/报价再切回来，步数会被重置。根因是 `validSteps` / `isValidStep` 校验数组只列了 6 个步骤，漏了 `delivery-check` 和 `export-archive`。前端 `technicalPlanStorage.ts` 与后端 `technicalPlanStore.cjs` 两处都漏，导致：(a) 持久化读取时整个 state 被判为非法 → 返回 null → 重置；(b) `updateStep('delivery-check')` 在 `applyPartial` 被 `isValidStep` 拦截 → 步数根本没存进库。

3. **商务标 / 报价 / 大纲是"写死的固定模式"**：改造前商务标/报价的表单字段、大纲生成的结构都是硬编码逻辑，没有读取招标文件解析出来的"投标文件组成 / 格式 / 要求 / 示例"。

**核心原则（用户硬性要求）**：始终以招标文件要求 / 示例 / 说明为最高优先级，硬编码逻辑只作兜底，冲突时以招标文件为准。

## 解决方案

整体改造分为两个阶段，本文档覆盖阶段一已落地内容，并界定阶段二边界。

### 总体流程（以目录为骨架）

```
上传招标文件 → 招标文件解析（含投标文件组成/格式）→ 目录生成（分水岭）
                                                          │
                                  ┌───────────────────────┼───────────────────────┐
                                  ▼                       ▼                       ▼
                              技术正文                   商务标                    报价
                         （全局事实/正文/扩写）    （资质/业绩/报价说明）      （明细/税费核算）
                                  │                       │                       │
                                  └───────────────────────┴───────────────────────┘
                                                          ▼
                                              交付检查 → 导出归档（完整标书合成）
```

目录（outline）生成完成后成为三块共享骨架。技术正文留在技术方案页内的现有 step 流程；商务标 / 报价通过编排区入口跳转到对应板块，并接收当前项目与招标解析上下文。

### 三块编排（阶段一：统一目录驱动骨架）

采用方案 A —— 在技术方案页内扩展为"目录驱动 3 块编排"。以 `TechnicalPlanHome.tsx` 为编排宿主，目录生成完成（`state.outlineData` 非空）作为分水岭，在 step 内容上方渲染编排区 `BidOrchestrationBand`（三张状态卡片：技术正文 / 商务标 / 报价）：

- **技术正文**：留在本页流程，根据进度跳转到 `global-facts` 或 `content-edit`；
- **商务标**：通过 `onNavigateSection('business-bid')` 跳转，页面接收 `currentProject` 与招标解析联动；
- **报价**：通过 `onNavigateSection('pricing')` 跳转，同样接收 `currentProject` 与招标解析联动。

阶段一以**渐进叠加**方式实现：编排区只在目录存在时出现，原有单线 step 流程完全保留，互不破坏。

### 状态机（技术方案 step）

8 个步骤构成线性状态机，是唯一真相来源（`TechnicalPlanHome.tsx` 的 `steps` 常量）：

```
document-analysis → bid-analysis → outline-generation → global-facts
→ content-edit → expand → delivery-check → export-archive
```

持久化校验（前端 `technicalPlanStorage.ts` 的 `validSteps`、后端 `technicalPlanStore.cjs` 的 `isValidStep`）必须与该列表一致，否则跨板块切换会丢步。本次已补齐两处缺失的 `delivery-check`、`export-archive`。

### 招标格式优先策略（大纲）

大纲生成新增"招标格式优先"路由，优先级高于既有评分对齐 / 自由生成：

1. **新增解析项 `bidFileStructure`（投标文件组成与格式）**：招标文件解析阶段从原文抽取"投标文件组成 / 格式 / 编制顺序"，作为大纲骨架依据。前后端解析项定义同步新增（`bidAnalysisTask.cjs` 与 `bidAnalysisWorkflow.ts`）。
2. **读招标原文 + bidFileStructure 渠道**：`outlineGenerationTask.cjs` 的 `runOutlineGenerationTask` 在加载技术方案后，读取招标文件原文（`workspaceStore.readTenderMarkdown()`）与 `bidFileStructure` 解析项内容。
3. **格式优先路由**：
   - 招标文件明确规定了投标文件组成 / 格式（`hasTenderFileStructure` 为真）→ 走 `tenderStructureWorkflow`：以投标文件组成为一级目录骨架，逐项补全二三级目录；
   - 未规定组成 → 回退既有 `alignedWorkflow`（评分对齐）/ `freeWorkflow`（自由生成）；
   - `tenderStructureWorkflow` 异常时 try/catch 强制回退 aligned/free，保证不阻塞主流程。

### 商务标 / 报价联动（全面联动）

- `AppRouter.tsx` 给 `BusinessBidPage` / `PricingPage` 传入 `currentProject`；
- 表单初值遵循"招标解析优先"：通过 `shared/utils/tenderLinkage.ts` 从技术方案状态（`technicalPlan.loadState()` 承载的 `bidAnalysisTasks`）抽取项目名、预算、质保期、响应时间、资格性审查、商务评分、交付与服务要求等，缺失时回退硬编码默认；
- 商务标持久化表单配置（此前 `sections:[]` 会丢弃表单），加载草稿时还原；
- 商务标 / 报价持久化 `bid_project_id`（后端 Store 已支持，默认回退当前项目）。

## 代码示例

招标格式优先路由（`outlineGenerationTask.cjs` 节选）：

```javascript
let outline;
if (hasTenderFileStructure(storedPlan)) {
  try {
    outline = await tenderStructureWorkflow(aiService, taskPayload, log);
  } catch (error) {
    log(`招标格式优先目录生成失败（${error.message || '未知错误'}），已回退到${taskPayload.mode === 'aligned' ? '评分对齐' : '自由'}生成模式。`);
    outline = taskPayload.mode === 'aligned'
      ? await alignedWorkflow(aiService, taskPayload, log)
      : await freeWorkflow(aiService, taskPayload, log);
  }
} else {
  outline = taskPayload.mode === 'aligned'
    ? await alignedWorkflow(aiService, taskPayload, log)
    : await freeWorkflow(aiService, taskPayload, log);
}
```

招标解析联动（`shared/utils/tenderLinkage.ts` 节选）：

```typescript
export function buildCommercialTenderSeed(state: TechnicalPlanState | null | undefined): CommercialTenderSeed {
  const projectInfo = parseJsonContent(getSuccessContent(state, 'projectInfo'));
  const delivery = parseJsonContent(getSuccessContent(state, 'deliveryAndServiceRequirements'));
  return {
    projectName: pickField(projectInfo, 'project_name'),
    projectBudget: pickField(projectInfo, 'project_budget'),
    warrantyPeriod: pickField(delivery, 'warranty_period'),
    responseTime: pickField(delivery, 'response_time'),
    afterSalesService: pickField(delivery, 'after_sales_service'),
    qualificationReview: getSuccessContent(state, 'qualificationReview'),
    businessScoring: getSuccessContent(state, 'businessScoring'),
  };
}
```

## 测试结果

- **大纲招标格式优先路由冒烟**：`client/scripts/outline-tender-format-smoke.cjs`，3 个场景全部通过：
  - 场景 1：招标规定组成 → 招标格式优先骨架（tenderStructureWorkflow）；
  - 场景 2：招标未规定组成（aligned）→ 回退评分对齐；
  - 场景 3：无组成解析项（free）→ 回退自由生成。
- **项目级模块隔离冒烟**：`client/scripts/project-scoped-modules-smoke.cjs` 通过（商务标 / 报价持久化无回归）。
- **类型检查与构建**：`npm run build`（tsc --noEmit + vite build）通过，无类型错误。

**阶段二低风险 3 项（v0.14.0 已落地）**：
- **编排区进度联动**：商务标 / 报价完成度回写编排卡片状态，替换硬编码 `'todo'`。`TechnicalPlanHome.tsx` 新增 `commercialBidStatus` / `pricingStatus` + useEffect，调 `commercialBid.list()` / `pricing.list()`（按当前项目自动过滤）推导 todo/active/ready。
- **大纲格式渠道深化**：`tenderStructureWorkflow` 补全二三级目录时也把 `bidFileStructure` 传给子目录生成。新增 `generateTenderStructureChildrenMessages` + `generateTenderStructureChildren`，循环中替换 `generateChildren`，受外层 try/catch 兜底保护。
- **导出三块一致性硬校验**：把软提示 warnings 升级为致命 / 重大 / 普通三级拦截（《完整标书合成与版式模板方案》§8）。`buildProjectExportPreview` 产出 `issues: ExportIssue[]`，`canExport = 无 fatal`；`ExportArchivePage` 按级分组展示，重大项导出前 `window.confirm` 确认。

**阶段三（v0.15.0 已落地）**：
- **招标驱动表单字段（商务标 + 报价）**：新增结构化 JSON 解析项 `bidFileStructureSchema`（`output:'json'`），与现有 `bidFileStructure`（markdown，面向大纲）互补。`buildFormSchema()` 读取解析结果与硬编码默认 schema 深合并，招标值优先。商务标资质初值由 schema 驱动（招标清单优先，否则回退后端默认类型）、业绩 Panel 显示招标最低数量要求、售后区叠加动态服务字段（`extraServiceFields`）；报价配置型字段（税率/币种/报价方式）由 schema 驱动默认值并标注招标建议值。核心明细表和后端生成逻辑不变。全程容错，解析失败/缺失均回退硬编码默认。

**阶段四（v0.16.0 已落地）**：
- **大纲格式渠道再深化**：OutlineItem 新增 `format_ref` 可选字段；新增 `parseFormatRefsFromBidFileStructure()` 正则解析 bidFileStructure 提取"条目标题→格式引用"映射；`tenderStructureWorkflow` 遍历一级条目时按标题匹配格式引用附加到 `format_ref`；`generateTenderStructureChildrenMessages` 当 `parentItem.format_ref` 存在时追加格式引用提示（如"招标要求本章按第六章格式1编写，请严格遵循该格式范本展开子目录"）；新增 `formatTemplateTables` 解析项（`output:'markdown'`），从招标原文提取格式范本表格正文，为后续内容生成阶段"按格式范本填充"提供数据基础。全程容错，解析失败回退现有行为。

**阶段五（v0.17.0 已落地）**：
- **内容生成消费格式范本**：`runContentGenerationTask` 从 `storedPlan.bidAnalysisTasks.formatTemplateTables` 读取格式范本内容，传递给 `buildChapterContentPlanMessages`（编排 prompt）和 `buildChapterContentMessages`（内容生成 prompt）。两个 prompt 函数在格式范本非空时注入用户消息，告知 AI 招标文件中的格式范本模板。内容生成 prompt 进一步检查 `parentChapters` 中的 `format_ref` 字段，如有则追加精细化提示（"本章节所属上级章节的格式要求：{format_ref}"），让 AI 按对应格式范本编写正文。全程容错，formatTemplateTables 缺失或为空时不注入，行为与现状一致。

**阶段六（v0.18.0 已落地）**：
- **编排区进度联动深化**：编排卡片新增 `detail` 字段，在 description 下方显示完成度详情。商务标从 `sections[0].form` 提取资质已填/总数、业绩已填/总数、生成状态（如"资质 3/5 · 业绩 2/3 · 已生成"）；报价从 items 提取已填报价项/总数、含税合计金额（如"报价项 5/8 · 含税 ¥125,000.00"）。detail 为空时不渲染，行为与现状一致。

**阶段七（v0.19.0 已落地）**：
- **导出校验深化**：`buildProjectExportPreview` 新增 `bidAnalysisTasks` 参数，从 `bidFileStructureSchema` 和 `projectInfo` 读取招标要求，新增 4 项深度校验——招标要求资质覆盖（major，缺少必备资质）、业绩数量达标（major，不足招标最低要求）、报价预算偏离（minor，超 30% 提醒）、资质关键字段缺失（minor，缺证书编号）。校验从"有没有"升级到"够不够"。附件管理和 OCR 确认待后续基础设施就绪后再加。
